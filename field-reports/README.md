# 实战经验库

这个目录保存真实 ChatCut 生产任务中出现的事实、失败链、产品问题、绕行方案和证据。它不是正式规则的堆放区，也不是只写成功经验的“战报”。

目标是解决两个问题：

1. 不用每出现一个新坑就发布新的 Skill 正式版本。
2. 下一次迭代不能从零开始，必须读取旧案例，并留下“读过以后做了什么决定”的记录。

## 目录契约

```text
field-reports/
  README.md
  iteration-log.md
  cases/
    YYYY-MM-DD-<case-slug>.md
```

- `cases/`：追加式原始案例。保留时间线、尝试、失败、修复、验证和未决问题。
- `iteration-log.md`：每次 Skill 迭代或命中案例标签的生产任务的读取回执。
- `references/`：从多个案例中晋升出的稳定做法。
- `rules/`：可机器执行、带反例和回滚条件的 hard policy。

## 开工读取协议

真实 ChatCut 任务或 Skill 迭代开始时：

1. 读取本文件。
2. 按 frontmatter 的 `tags`、`chatcut_surfaces`、`failure_signatures` 找到相关案例。
3. 完整读取案例，不能只读执行摘要。
4. 把读取结果写入 `iteration-log.md`：
   - 读取了哪些案例；
   - 哪些结论本次采用；
   - 哪些结论因版本、证据或任务差异被拒绝；
   - 新增了什么 canary；
   - 哪些结论有资格晋升。

## 收工写入协议

任务结束时：

1. 新事实优先追加到已有案例；不同根因或不同产品面才新建案例。
2. 记录实际结果，不把计划写成结果。
3. 区分证据等级：
   - `E0`：推测或代理内部判断，没有用户/像素/回读证据。
   - `E1`：结构回读、工具返回、DOM/日志。
   - `E2`：用户在编辑器的可见观察，或本地源文件抽帧/试听。
   - `E3`：ChatCut 合成帧、最终导出文件、可复算媒体指标。
4. 记录 `expected / actual / impact / workaround / requested_fix`，让产品经理和开发可直接转为 issue。
5. 记录证据缺口。没有保存编辑器截图时，要明确写“用户观察，未留像素附件”，不能冒充 E3。

## 规则晋升

一条案例结论进入 `references/` 至少满足：

- 两个独立案例，或同一问题在不同资产/实例上重复；
- 有 E2/E3 证据；
- 有适用范围和反例；
- 有失败时的回滚动作；
- 不与官方 ChatCut skill 和当前工具 schema 冲突。

进入 `rules/` 还必须：

- 可机器判定；
- 有正/反 fixture；
- 有稳定 rule ID；
- 有 owner、回滚和版本漂移策略。

## 修订与纠错

- 案例原则上不覆盖旧结论。
- 事实错误可原位修复，但必须在“修订记录”说明原因。
- 产品版本变化使旧结论失效时，新增 `supersedes` 案例，并在旧案例顶部链接新案例。
- 不删除仍有历史解释价值的失败路径。

## 公开仓脱敏

禁止提交：

- ChatCut 项目 ID、资产 ID、签名 URL、短期导入 token；
- 用户本机绝对路径；
- 原始字幕全文、人物画面和私有截图；
- 品牌私词表、真实业务敏感数字；
- OAuth、API key、邮箱、手机号等标识。

允许提交：

- 匿名化帧数、时长、轨道角色和 source offset；
- 错误字符串、工具行为、验证流程；
- 脱敏标题和通用 ASR 错误模式；
- 对 ChatCut 产品团队有复现价值的 expected/actual。
